## Brief
Build the beating heart of touchRPG's P0 vertical slice: **the tap-parry loop**, in Unity, portrait.
The GDD (`docs/spec/00-gdd-v0.3.md` — **v0.3 is current**; the build ran against v0.1, whose P0-A
sections are unchanged in v0.3) is the **single source of truth** — read §0, §2, §4, §6.1-6.2 and
§7.2 before writing code. Where this brief and the GDD disagree, **the GDD wins**.

P0 exists to answer exactly one question (GDD §10): **"터치 패링이 손맛이 있는가"** — does the touch
parry actually feel good? Everything here serves that. Do not broaden scope beyond it.

### Build
1. **Portrait-fixed** scene (GDD §1; landscape is a non-goal), 4-layer combat screen per GDD §6.1:
   정보층 (monster name + HP bar showing the 70%/35% phase ticks) / 몬스터층 (monster; parry markers
   **anchored to monster parts** — MUST) / 전장층 (player, ground) / 파티층 (portrait slots; solo = 1,
   a stub is fine).
2. **Input** (GDD §4.1): IN-1 tap monster part = basic attack; IN-2 tap parry marker = parry; IN-4 tap
   bare ground = move. Implement the **§4.2 priority resolution exactly**: parry marker > dodge zone >
   party portrait > monster part > ground — no exceptions; it is the core mis-input guard.
   (IN-3/IN-5/IN-6/IN-7 belong to later tasks — leave clean seams, do not build them.)
3. **Parry marker** (GDD §6.2): yellow double ring, outer ring contracting into the inner; the instant
   they overlap = perfect. **The contraction speed IS the rhythm information.** Perfect = gold burst,
   good = white. The marker anchors to the monster part, not to the screen.
4. **Judgment windows** (GDD §4.3/§12): perfect ±0.15s, good ±0.35s. **MUST be externalized** to a
   ScriptableObject/config — hardcoding a gameplay constant is a MUST NOT (GDD §0).
5. **Combo multiplier** (GDD §4.4): +1 per perfect; good maintains but does not raise; reset to 0 on hit.
6. **람팡 pattern P1 — 도토리 투척** (GDD §7.2): class C-1, yellow ring on the cheek pouch, **parry ×2**,
   ±0.35s, 정박 2연 (on-beat double — the learning pattern), phase 1+, failure = small damage. Drive it
   from a **data-defined pattern sheet** (ScriptableObject), never hardcoded logic — every later 람팡
   pattern (P2-P7) and every future monster must slot into the same structure without a rewrite.
7. **Placeholder art is expected** (GDD §0 grants placeholder art to team discretion): a chunky cute
   squirrel silhouette with visible cheek pouches; primitives are fine. Spend the time on *feel*, not
   on art fidelity.

### The TBD rule (MUST — treat this as a gate, not a nicety)
GDD §13 leaves **TBD-1 (combo cap)** and **TBD-2 (damage curve)** deliberately undecided — the director
sets them *after* the P0 playtest. Therefore:
- Implement the mechanism, externalize both values, and label them **explicitly as provisional
  TBD-1 / TBD-2 placeholders in the config**.
- **MUST NOT** present them as decided, and MUST NOT invent any other missing gameplay number
  (judgment, damage, drop rates). Report every provisional value used.

The GDD names this failure mode in its own words: *"그럴듯한 보간(hallucinated design)은 이 프로젝트에서
가장 경계하는 실패 모드다."* An invented design decision fails this task however good the result looks.

## Acceptance criteria (observable — the 팀장 scores against these)
- [ ] App runs **portrait-fixed**; the 4 layers of GDD §6.1 are present in roughly the specified
      proportions (10% / 10-60% / 60-85% / 15%).
- [ ] Tapping the parry marker at ring-overlap gives **perfect** (gold burst); slightly off gives
      **good** (white); far off = miss + damage + combo reset. Demonstrable in a real run.
- [ ] Perfect ±0.15s and good ±0.35s are **read from config**, not code — editing the config changes the
      behavior (demonstrate this).
- [ ] Priority §4.2 holds: with a parry marker over the monster body, the tap **parries** and does not
      trigger a basic attack; bare ground taps move.
- [ ] Combo multiplier rises +1 per perfect, holds on good, resets to 0 on hit.
- [ ] 람팡 P1 도토리 투척 fires as a 2-hit on-beat pattern with cheek-anchored yellow markers at ±0.35s,
      and is defined in a **data asset** (show it), not in code.
- [ ] Gameplay colour on screen stays inside the GDD's fixed channels (here: yellow parry, gold reward).
      No invented gameplay colours.
- [ ] Zero C# compile errors; any tests pass; **no Unity editor left holding the project lock**.
- [ ] Every provisional/TBD value is listed in the report and marked not-decided.
