## Brief
Turn the pattern library T001+T002 built (도토리 투척 + P2-P7, all currently only dev-triggerable in
isolation) into an actual **playable hunt from start to finish**: real HP-driven phase transitions,
phase-weighted pattern selection, and a hunt-complete end state. This is the task that makes P0
genuinely completable solo — the precondition for actually validating GDD §10's question
(*"터치 패링이 손맛이 있는가"*). The GDD (`docs/spec/00-gdd-v0.4.md`) is the single source of truth —
read §5.1-5.3, §6.1 before writing code. Where this brief and the GDD disagree, the GDD wins.

**You are replacing/extending a known seam, not building from nothing.** `MonsterPatternPlayer.cs`
already has a `currentPhase` field and its own doc comment: *"The repeat loop below... is a P0 demo
driver only, NOT the GDD §5.1 three-phase session system - that is a separate, larger task."* This is
that task. `TriggerPatternById` (the dev/QA independent-trigger escape hatch) and
`AutoPlayToggleButton`/`PatternTriggerButton` (dev tools) should keep working for QA — do not remove
them — but the **default play path** (no dev override) must run the real phase system below.

### Build
1. **Real HP-driven phase transitions** (§5.1 table): phase 1 = monster HP 100-70%, phase 2 = 70-35%,
   phase 3 = 35-0%. Drive this from `MonsterHealth`'s actual current HP (not a manually-set
   `currentPhase` field) — when a hit brings HP across 70% or 35%, the phase advances immediately, mid-
   pattern-cycle if needed. T001 already built the HP-bar phase-tick display (§6.1 정보층) — wire it to
   these real transitions if it isn't reading live phase state already.
2. **Phase-weighted pattern selection**, replacing `DriveLoop`'s current cycling with a scheme that
   honors §5.1's composition per phase:
   - **Phase 1**: C-1 패턴 ~70%, C-2 패턴 ~30% of picks. **금지: 릴레이(C-3)·페이크.** Also: P2 꼬리치기
     (its 엇박/반박자 element) must NOT appear yet — §5.1 says off-beat variants only "등장" starting
     phase 2, implying phase 1 stays on-beat (도토리 투척 P1 only, for C-1).
   - **Phase 2**: 엇박 variants unlock (P2 꼬리치기 becomes eligible). **C-3 릴레이 (P5) unlocks — but
     only once this phase**, with **완화된 실패 페널티** (per §5.2's easy-monster rule: only the player
     who missed their beat is hit + only their own combo resets, not the whole party — trivial for solo
     since party=1, but implement the reduced-penalty branch, not the harsh one). Fakes (P4) still
     forbidden.
   - **Phase 3**: 2-3 relay (P5) occurrences, C-5 (P6) present, **페이크 해금** (P4 becomes eligible),
     maximum pattern density (shortest gaps between beats within what's already tuned).
   - The exact selection algorithm (weighted random, a scripted queue, etc.) is your call (GDD §0 grants
     architecture to team discretion) — but the phase-eligibility gates above (what CAN appear each
     phase) are MUST, not suggestions.
3. **Guaranteed groggy rush per phase transition** (§5.1 MUST): *"페이즈 전환마다 그로기 러시(C-4)를
   최소 1회 보장한다."* P7 (그로기, C-4) is only reachable via a successful P5 relay chain (already true
   from T002). Since phase 1 forbids relay, the earliest possible groggy rush is in phase 2. Guarantee,
   don't leave to chance: near each phase's start (entering phase 2, entering phase 3), force/prioritize
   a relay attempt so a groggy rush demonstrably happens at least once per phase before that phase ends.
   State your chosen mechanism in the report (e.g. a forced relay injection vs. a pity-counter) — either
   is fine, just make it actually guaranteed, not merely more likely.
4. **Solo run to completion**: when `MonsterHealth.CurrentHP` reaches 0, the hunt ends — stop the
   pattern-drive loop cleanly (no orphaned coroutines/markers), show a clear hunt-complete state (a
   simple result panel is enough; no need for GDD §9-style rewards UI, that's out of P0 scope), and make
   it possible to start a fresh run afterward (re-enter combat scene or reset state — your call, just
   don't leave the game in a dead/uninteractable state).
5. Keep everything from T001/T002 untouched in its own right: judgment windows still config-driven,
   gameplay colour channels still fixed at 4, `TapPriority`/`CombatInputController` unchanged, no new
   input vocabulary.
