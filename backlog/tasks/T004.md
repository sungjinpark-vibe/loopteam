## Brief
Close the remaining gaps in GDD §6.2's marker visual language — specifically the **relay marker**,
which currently reuses the plain parry-marker look instead of its own specified visual. The GDD
(`docs/spec/00-gdd-v0.4.md`) is the single source of truth — read §6.1-6.2 (and §5.2 for relay behavior
context) before writing code. Where this brief and the GDD disagree, the GDD wins.

**Scope is narrow and explicitly bounded to §6.1-6.2** — this is P0's definition (`VISION.md` §2: *"P0 =
... combat UI §6.1-6.2"*), not §6.3 (party-layer interactions like cover warnings). **Do NOT build IN-7
(cover action)** — it stays the explicit out-of-scope seam T001 already left (`PartyPortraitSlot.OnTapped`
is a documented no-op). Do not add real multi-party relay ordering/UI — party is still solo (1), a stub,
per T001-T003. This task is a **completeness/polish pass**, not new mechanics.

- Dodge zone (blue circle + guide line + clockwise gauge) — T002.
- Rush zone (weak-point circle + tap ripples + progress gauge + finish flourish) — T002.

### What's missing (build)
1. **Relay marker's own visual** (§6.2 MUST): *"릴레이 마커: 붉은 링 + 순번. 내 차례: 불투명 + 테두리
   펄스 + 햅틱 (3중 신호, MUST)."* Currently `ExecuteC3Relay` (in `MonsterPatternPlayer.cs`) spawns plain
   `ParryMarker` instances for the solo 2-3 tap consecutive sequence (§5.2's solo substitute) — same
   yellow look as a normal parry, not red, no sequence number. Add a distinct relay-marker visual mode
   (red ring per `GameplayColors.Relay`, a sequence number badge showing which tap in the sequence this
   is (1/2/3), and — since solo is always "current turn" — the full triple signal every relay beat:
   opaque rendering + a pulsing border + a haptic trigger). You may extend `ParryMarker` with a relay
   mode/variant rather than writing a parallel class, if that keeps judgment logic (which must stay
   identical — same config-driven windows, same `JudgmentEvaluator`) from forking. Haptic: use
   `Handheld.Vibrate()` guarded for editor/non-mobile (no-op there is fine, document it) — this cannot be
   felt in an Editor/PC test run, so say so plainly in your report rather than claiming it was verified.
2. **Relay success light-beam** (§5.2/§6.3 MUST, but scoped here to the solo relay flow only — not a
   general party system): *"성공 연출은 파티원 초상 → 몬스터로 빛이 이어지는 흐름."* On a successful
   solo relay sequence, show a brief light/beam effect from the party portrait slot (bottom, §6.1
   파티층) toward the monster. Simple is fine (a UI line/particle that fades, matching this project's
   existing placeholder-art bar) — this is presentation, not a new mechanic.
3. **Audit pass**: walk every marker type against §6.2's text line by line and report any other drift
   found (do not silently fix things outside relay — report them, since anything beyond the two items
   above needs this task's own acceptance criteria to cover it, and inventing new scope mid-task is what
   this project's discipline exists to prevent). If nothing else is found, say so explicitly.
