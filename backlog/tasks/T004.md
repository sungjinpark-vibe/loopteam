## Brief

Produce the **art design system** for the Life Town Unity rebuild — concrete enough that `client-dev`
can build the App-layer screens (the village especially) without asking a visual question. This unblocks
everything visual: the village is the app's payoff screen ("눈에 보이게"), and nothing in `LifeTown.App`
can be built until its look is specified.

This is a **rebuild, not a blank-page invention.** The Flutter original has 10 art docs
(`docs/design/00-design-system.md` … `10-art-upgrade-plan.md`) and a locked visual identity. Read them,
carry forward what holds, and adapt it to Unity (isometric Tilemap + URP 2D lighting), which the Flutter
version dropped. Do not restart the identity.

### Locked by the spec (do NOT reopen — these are constraints, not choices)
- **Cozy pastel + coquette**, healing tone. Mongsil the mascot exists.
- The **Board A palette is `절대 유지`**; the **7 category colours are shipped hex constants** (already in
  `Assets/LifeTown.Economy.Core` config from T002 — match them exactly).
- **In-house geometric / isometric** (D6): flat colour, clean shapes, **128×64 diamond tiles**, 3 tier
  stages per building. No commissioned illustration — code/geometry-producible assets only.
- Life-logging identity: growth and leisure categories are **visually equal** — no category is styled as
  "more virtuous." (§6.4 leisure-parity constraint.)

### What to produce
- The **design system**: color roles (from the locked palette), typography, spacing, component styles
  (buttons, cards, the session receipt, progress). Concrete values (hex, px).
- **Per-screen visual design** for the 7 MVP screens (spec §8): village, timer flow (category-select /
  running / result), stats, settings, onboarding — layout, key elements, states.
- The **village visual language**: how a building reads its category (colour) and level (tier stage),
  how the isometric grid composes, day/night lighting treatment (spec §6.2).
- An **art-order spec** for anything code cannot produce, sized and counted, so the PM knows the gap.

## Acceptance criteria (아트팀장 scores A1-A5, pass 90)
- [ ] Carries the locked identity forward faithfully (palette, 7 category hex, cozy-pastel, Mongsil) —
      A verifiable against the source docs and the Core color constants.
- [ ] Concrete enough that client-dev can build a screen without a follow-up question (hex, px, states).
- [ ] The village reads at a glance: category legible by colour, level legible by tier stage.
- [ ] Exploits Unity where it helps (isometric Tilemap, Light2D day/night) and is honest where it does not.
- [ ] Leisure-parity respected: no category styled as more/less virtuous.
- [ ] Marks assumptions; lists any visual decision that is genuinely the director's to make.
